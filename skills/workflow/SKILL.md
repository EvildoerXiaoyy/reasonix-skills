## Development Workflow — 四层架构 v2.0

**设计原则**：契约驱动解耦、人类把关关键决策、Agent 行为可观测、流程可灰度裁剪。

当你被调用时，先展示工作流总览，然后询问用户当前处于哪个阶段，引导他们进入对应步骤。

### 阶段详解

#### 阶段一：整体规划（一次性）

**目标**：定架构、定边界、定契约，全局只做一次，完成后由人类确认基线。

| 步骤 | 命令 | 说明 |
|------|------|------|
| 架构蓝图 | `/arch-workflow` | 输出必须包含：组件图、数据流、Scope Challenge、ADR 骨架、模块依赖顺序，以及形式化契约文件 |
| 产出 PRD | `/to-prd` | 结构化需求文档 |
| 想法拷问 | `/grill-me` | 逐条追问方案细节，发现漏洞 |
| 工程评审 | `/plan-eng-review` | EM 视角审计划，确定模块边界，生成评审报告 |
| **👤 人类签字** | **人工确认** | **架构师/EM 必须显式确认基线。未签字，阶段二不得启动** |

**产出物**：ARCHITECTURE.md + PRD + API_CONTRACT.yaml + DB_SCHEMA.sql + MODULE_DEPENDENCY.md + 审查报告 + 人类签字记录

**阶段 commit**：`git commit -m "stage-1: 整体规划完成 — <项目名>"` — 锁定规划基线，后续阶段以此为基础。

**关于基线**：基线在人类签字后生效。阶段二中若发现基线设计不合理，可通过 `/amend-contract` 发起轻量级修订，而非被动等待阶段四的漂移检测。

---

#### 阶段二：模块循环（契约驱动，支持有限并行）

**目标**：每个模块独立开发，人机各司其职。充分利用阶段一的契约成果，允许无依赖阻塞的模块并行推进。

**循环模板（每个模块执行一遍）**：

**① 前置确认**
- 执行 `/dep-status` 读取 MODULE_DEPENDENCY.md，确认所有前置依赖模块已通过门禁。
- **若依赖未就绪**：自动调用 `/mock-gen` 基于 API_CONTRACT.yaml 生成轻量 Mock/Stub，允许当前模块进入 TDD 循环。Mock 行为写入 `MOCK_BEHAVIOR.md` 以供后续真实替换时参考。

**② 测试接力（契约驱动）**
- 调用 `/test-gen`，AI 基于 API_CONTRACT.yaml 和 PRD 生成测试框架：
  - 测试类骨架、Mock/Stub 初始化代码
  - 基础 Happy Path 测试用例
  - 用 `// TODO:` 预留边界条件、异常处理、并发安全、极限数据等测试方法占位符
- **👤 人类介入**：审查 AI 生成的测试，并在 TODO 占位符处补充核心 Edge Case 和异常流。
- **/review-request**：高风险模块（安全相关、核心业务、外部接口）必须执行二审；低风险模块可选。
- 测试定稿，代码入库，同时输出 `TEST_INTENT.md` 记录每个测试的防御意图，作为后续 review 的仲裁依据。
- **`git commit -m "test: <模块> 测试定稿"`** — 测试文件入库，此时运行测试应全部失败（Red）。
- ⚠️ 若后续实现阶段修改测试涉及契约变更，必须执行 `/amend-contract`，不可静默修改。

**③ TDD 实现循环（含健康度监控）**
- **Intent 强制输出**：每次调用 `run_command` 前，AI 必须输出结构化 Intent：
  ```yaml
  goal: "解决 xxx 测试失败"
  hypothesis: "根因是 xxx"
  planned_actions: [read_file, grep, run_command]
  expected_outcome: "测试通过"
  ```
- 红绿循环：运行测试(Red) → 编写最小实现(Green) → 进入下一轮，**每完成一个功能点（如一个方法的所有测试通过）后 `git commit -m "feat: <模块> <方法> 通过"`**。
- 该模块全部测试用例通过后 **`git commit -m "feat: <模块> 全部功能完成"`** — TDD 实现阶段结束。
- 重构：所有测试通过后，调用 `/refactor` 在无副作用前提下优化代码结构。
- **熔断机制（增强版）**：
  - 传统触发：同一测试用例连续失败 ≥3 次
  - **新增**：健康度指标监控（文件修改频率、信息收集密度、Surprise 因子累计）
  - 触发后**不是强制进入 /debug，而是暂停并提示开发者**：
    > "同一用例已连续失败 3 次。建议执行 /debug 诊断。是否继续？[Y:继续实现/N:进入诊断/C:人工检查代码]"
- 遇架构级决策调用 `/adr` 记录。
- **Commit 规范**：遵循以下类型，每完成一个逻辑步骤即 commit，不可批量积攒：
  - `test: <模块> 测试定稿` — 测试文件入库
  - `feat: <模块> <功能点> 通过` — 一个功能点的所有测试通过
  - `feat: <模块> 全部功能完成` — 模块内所有用例通过
  - `refactor: <模块> <动作>` — 重构后
  - `adr: <标题>` — 新增架构决策
  - `gate: <模块> 门禁通过` — 模块门禁
  - `stage-N: <阶段名> 完成 — <项目>` — 阶段结束
- **契约修订**：若实现中发现 API 契约或类型定义需要调整，执行 `/amend-contract`：
  - 记录变更原因和影响分析
  - 更新契约文件
  - 标记所有受影响模块为"需重新验证"

**④ 模块门禁（阻塞式）**
- `git commit -m "gate: <模块> 模块门禁通过"` 记录门禁状态。
- 运行测试套件，覆盖率达标。
- `/codereview --gate` 执行阻塞式代码审查（静态分析 + 规范检查 + 人工 review 流程）。
- 执行契约兼容性冒烟测试（使用真实依赖或 Mock 验证接口行为一致性）。
- ✅ 通过后标记模块完成，模块状态写入 MODULE_STATUS.md。

**产出物（每个模块）**：通过的测试 + 实现代码 + 增量 ADR + 可能更新的契约文件 + `git commit -m "gate: <模块> 门禁通过"`

---

#### 阶段三：集成（持续集成，非最后一次性）

**目标**：确保模块间协作正确。模块完成即进行局部集成，而非等待全部完成。

| 步骤 | 命令 | 说明 |
|------|------|------|
| 局部集成 | (手动/自动化) | 每个模块通过门禁后，立即与已就绪模块进行接口对齐冒烟 |
| 全量测试 | (手动) | 全部模块完成后运行完整测试套件 + 端到端集成测试 |
| 跨模块 review | 手动 | 检查模块间接口是否对齐、数据流是否完整、有无未处理的 TODOs |
| 合并门禁 | `/codereview --gate --base main` | blocker / waiver / recheck 全流程，决定能否合并 |

**产出物**：门禁报告 + 集成测试结果 + `git commit -m "stage-3: 集成完成 — <项目名>"`

---

#### 阶段四：回顾

**目标**：沉淀知识，修复文档漂移，生成高质量交接物。

| 步骤 | 命令 | 说明 |
|------|------|------|
| 漂移检测 | `/reconcile` | 对比最终代码 vs ARCHITECTURE.md、PRD、API_CONTRACT，**生成漂移报告（含差异项和修复建议），不自动修改** |
| **👤 人类裁决** | **人工确认** | **开发者逐条审核漂移项，决定"代码迁就文档"或"文档迁就代码"** |
| 决策沉淀 | `/adr` | 回顾和定稿所有 ADR |
| 上下文压缩 | `/handoff` | 生成交接文档，包含：核心架构决策摘要、未解决问题、Context Budget 建议（关键文件列表 + 优先级） |

**产出物**：漂移报告 + 人类裁决记录 + ADR 定稿 + handoff 文档 + `git commit -m "stage-4: 回顾完成 — <项目名>"`

### 根据场景裁剪

| 场景 | 建议 |
|------|------|
| 小改动 / bugfix | 跳过整体规划，直接 TDD 循环 → 全量测试 → `/codereview --gate`。务必补最小化 ADR |
| 单模块项目 | 跳过模块循环依赖检查和 Mock 生成，一个模块做完直接进集成 |
| 实验性功能 | `/grill-me` + 简化 TDD，可跳过 `/review-request`，但保留 `/codereview --gate` |
| 多人并行开发 | 充分利用 `/mock-gen` + 契约解耦，各模块独立推进，通过 MODULE_STATUS.md 同步进度 |
| 上下文紧张的长项目 | 阶段四 `/handoff` 必须产出 Context Budget 建议，标记关键文件，避免下次会话加载冗余信息 |

### 工具权限说明

- **顾问模式**：仅将 AI 作为聊天指导，`allowed-tools: ask` 即可。
- **自主运行模式**（如在 Cursor/IDE Agent 中）：建议配置 `read_file, list_dir, run_command, glob, grep`，以实现测试运行、门禁检查等自动化流程。
- **安全约束**：生产环境建议限制 `run_command` 的白名单（如仅允许 `npm test`, `go test`, `pytest`, `make lint` 等），降低注入风险。

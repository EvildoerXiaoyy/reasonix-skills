# Safe Refactor — Green Phase Cleanup

遵循 TDD 的 **Red-Green-Refactor** 循环，本 skill 在测试全部通过（Green）后执行，专注于无副作用的代码清理。

## 前置条件

- 当前模块的所有单元测试 **必须已通过**（本 skill 会先运行一次测试套件确认，若失败则报错并退出）。
- 已存在测试文件（否则重构无安全保障）。

## 重构检查项（优先级从高到低）

1. **消除魔术数字与字符串**
   替换为命名常量或枚举，并补充注释。

2. **提取重复逻辑**
   发现重复代码块（>5 行相似）提取为私有方法或工具函数，保持接口不变。

3. **改善命名**
   - 变量、函数、类命名是否准确反映其职责？
   - 缩写是否被广泛理解？
   对于含义模糊的名字，提出重命名建议（并确保在测试中更新引用）。

4. **降低圈复杂度**
   将过长的函数（>30 行）拆分为多个小函数；将深层嵌套的条件分支改为早返回或策略模式。

5. **优化代码布局**
   按逻辑对方法排序（如构造-公共-私有），让相关代码靠近。

6. **添加注释（仅针对"为什么"，而非"做什么"）**
   对于复杂的业务逻辑或非显而易见的实现决策，若缺少注释，补充简要解释。

7. **可观测性检查（RPC 边界模块）**
   如果当前模块是 gRPC/HTTP handler、消息消费者或状态机，检查 Trace ID 是否在关键路径中正确透传、关键业务路径是否有 OTel Span 埋点。若缺失，添加 `// TODO: observability` 占位。

## 工作流

1. **运行全量测试**
   执行测试套件，确认所有测试通过。若有失败，直接终止并提示先修复。

2. **识别重构点**
   扫描模块代码，输出待优化项清单（带文件和行号），展示清单并等待用户确认。

3. **逐项重构**
   对每一项：
   - 修改代码
   - 立即运行相关测试，确保绿色
   - 如果测试失败，回退并分析原因（若连续失败 ≥3 次，触发 `/debug` 诊断）
   - 提交（推荐使用 Conventional Commits 格式如 `refactor: extract validation logic`）

4. **最终验证**
   全量测试通过后，生成简要的重构报告，包含修改列表和改善点。

## 输出

- 重构报告（可内联在对话中）
- 建议的 commit message

## 交接

告知用户重构完成，可以继续模块门禁或进入下一模块。

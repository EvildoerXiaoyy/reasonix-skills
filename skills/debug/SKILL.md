# Structured Debugging — 熔断诊断模式

当同一个测试在多次尝试后仍然失败（默认阈值 3 次），说明单纯的试错已失效。本 skill 启动一个规范的排查过程，避免陷入"改了东边坏了西边"的恶性循环。

## 触发条件

- 用户手动调用 `/debug`，或
- 在 TDD 实现循环中，同一测试连续失败 ≥ 3 次（由工作流或 `/refactor` 自动建议）。

## 诊断步骤（严格按顺序）

### 1. 收集信息
- 获取完整的错误日志或测试失败输出。
- 读取相关源文件（被测函数、测试代码），不修改任何内容。
- 回顾最近的 git diff（若为回归测试），定位可能引入问题的变更。

### 2. 提出假设（至少 2 个）
基于代码和错误信息，给出可能的根因假设。例如：
- 假设 A：并发竞态条件导致数据不一致
- 假设 B：Mock 对象未正确配置，返回了意外值
- 假设 C：边界条件未处理，传入空值时发生 panic

### 3. 设计验证方案（优先使用日志或小验证代码）
为每个假设设计一个微验证方案：
- 在关键位置加 `print`/`log` 语句，输出运行时变量的值
- 编写一个最小独立的验证脚本（5-10 行），仅测试该假设
- **如果是跨服务调用或包含外部依赖的错误，优先通过注入或提取 Trace ID 来追溯上下文链路，或检查相关 Metrics 面板，避免仅依赖单点日志盲目推断**
- 运行该方案并分析输出

严禁直接修改业务逻辑。

### 4. 定位根因
根据验证结果，确定最可能的根因。如果一次验证不足以确认，继续下一轮假设，但最多进行 3 轮诊断。若仍无法定位，建议以下方向：
- 请求同伴 review（或调用 `/review-request` 审视该代码段）
- 缩小问题范围，尝试用更简单的测试重现

### 5. 修复与回归测试
根因确定后，修改代码。修改后：
- 运行原先失败的测试，确认通过
- 运行全量测试套件，确保无回归
- 若有必要，为发现的根因类型补充针对性测试用例

## 输出

每次诊断会话应输出一个结构化的诊断报告：

```markdown
## Debug Report — <timestamp>

**失败测试:** `TestPaymentTimeout/retry_exhausted`
**错误信息:** `expected "retry" but got "fail"`

**假设 A:** … （证实 / 排除）
**假设 B:** … （证实 / 排除）

**根因:** 重试计数器的竞态条件，在 go routine 间共享未加锁。

**修复:** 为 counter 添加 sync.Mutex 保护。

**回归:** 全量测试通过，新增并发测试 `TestPaymentTimeout/concurrent_retries`。
```

## 交接

将诊断报告和修复方案呈现给用户，建议视情况调用 `/adr` 记录此次发现的架构级缺陷。
